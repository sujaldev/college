    % --------------------------------------
    \begin{frame}[t]{Fixed Partitioning: Concept}
        Fixed partitioning divides physical memory into a set of
        \textbf{predefined, fixed-size partitions}.

        \medskip

        Key points:
        \begin{itemize}
            \item Number and size of partitions are decided at system boot.
            \item Each partition holds at most one process at a time.
            \item A process is loaded into the smallest partition large enough to contain it.
        \end{itemize}

        \medskip
        This is the simplest form of contiguous memory allocation.
    \end{frame}

    % --------------------------------------
    \begin{frame}[t]{Fixed Partioning: Drawbacks}
        Fixed partitioning suffers from \textbf{internal fragmentation}:
        unused memory \emph{inside} a partition.

        \medskip

        Why it occurs:
        \begin{itemize}
            \item Processes rarely match the partition size exactly.
            \item Extra space inside the assigned partition cannot be used by others.
        \end{itemize}

        \medskip
        Example: a 40\,KB process placed in a 64\,KB partition wastes 24\,KB.
    \end{frame}


    % --------------------------------------
    \begin{frame}[t]{Fixed Partioning: Drawbacks}
        Additional limitations:
        \begin{itemize}
            \item Total number of simultaneously running processes
            is capped by the number of partitions.
            \item Memory distribution cannot adapt to real workload changes.
            \item Large processes may not fit any partition even if total free
            memory is sufficient.
        \end{itemize}

        \medskip
        These constraints made fixed partitioning unsustainable as programs grew larger and more varied.
    \end{frame}


    % --------------------------------------
    \begin{frame}[t]{Fixed Partioning: Summary}
        Fixed partitioning is simple but fundamentally rigid:
        \begin{itemize}
            \item Wastes memory (internal fragmentation).
            \item Artificially limits multiprogramming.
            \item Cannot respond to dynamic workload requirements.
        \end{itemize}

        \medskip
        Dynamic partitioning was introduced to escape these limitations.
    \end{frame}

    \begin{frame}[t]{Dynamic Partitioning: Concept}
        Dynamic partitioning allows the operating system to allocate memory blocks
        \textbf{of variable size}, matching the exact needs of each process.

        \medskip

        Key ideas:
        \begin{itemize}
            \item Memory begins as one large free block.
            \item When a process arrives, the OS finds a free region large enough.
            \item The process receives a block equal to its required size.
            \item On exit, its block is returned to the free list.
        \end{itemize}

        \medskip
        This eliminates the internal fragmentation seen in fixed partitioning.
    \end{frame}

    \begin{frame}[t]{Dynamic Partitioning}
        Fixed partitioning wastes memory through \textbf{internal fragmentation}:
        unused space inside rigid, predefined partitions.

        \medskip

        Dynamic partitioning was created to solve this problem by:
        \begin{itemize}
            \item Allocating memory exactly as needed.
            \item Avoiding fixed-size, pre-cut partitions.
        \end{itemize}

        \medskip
        The goal: give processes precisely the memory they request.
    \end{frame}


    % --------------------------------------

    \begin{frame}[t]{Dynamic Partitioning: External Fragmentation}
        Although dynamic partitioning avoids internal fragmentation,
        it introduces \textbf{external fragmentation}.

        \medskip

        As processes are created and terminated:
        \begin{itemize}
            \item Free memory becomes scattered into small, non-adjacent holes.
            \item Total free memory may be enough for a new process, but
            no \emph{single} contiguous block may be large enough.
        \end{itemize}

        \medskip
        This is the fundamental limitation that eventually pushed OS design
        toward paging.
    \end{frame}


    % --------------------------------------

    \begin{frame}[t]{Placement Algorithms}
        Because free memory holes vary in size, the OS must choose where to place new processes.

        \medskip

        Common strategies:
        \begin{itemize}
            \item \textbf{First-fit}: use the first hole large enough.
            Fast and works well in practice.
            \item \textbf{Best-fit}: use the smallest hole that fits.
            Minimizes leftover space per allocation but increases fragmentation.
            \item \textbf{Worst-fit}: use the largest hole.
            Attempts to keep holes large; performs poorly overall.
        \end{itemize}
    \end{frame}


    % --------------------------------------

    \begin{frame}[t]{Compaction}
        To reduce external fragmentation, the OS may perform \textbf{compaction}:
        \begin{itemize}
            \item Move allocated blocks to one side of memory.
            \item Combine scattered free holes into one large block.
        \end{itemize}

        \medskip

        Limitations:
        \begin{itemize}
            \item Requires relocating processes.
            \item Expensive and disruptive.
        \end{itemize}

        Compaction is a temporary fix, paging eventually replaced this entire model.
    \end{frame}
